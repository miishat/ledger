# v0.7.2-beta Design

Date: 2026-07-18
Status: Approved by Mishat

Ten items: two features (shared expenses, PortfolioAnalyst import), budget reporting upgrades, and a batch of bug/UX fixes. Minimal testing per user instruction: unit tests for new parsers/selectors only.

## 1. Total monthly budget visibility

Problem: categories have `targetAmount` but nothing sums them anywhere.

- `src/store/budgetSelectors.ts`: new selector returning total budgeted (sum of expense-category `targetAmount` where > 0) and total spent for the selected period.
- `BudgetProgressWidget` ("Budget vs. Actual"): header row above the category list showing `Total: {spent} / {budgeted}` with its own progress bar, styled like category rows but bolder. Pace label follows the same >110% / <90% rules.
- `CategoryManagerWidget` (Setup tab): show the summed monthly target under the expense groups.
- Income categories are excluded from the total.

## 2. Shared expenses (split + IOU tracking)

Problem: no way to record bills paid for others (group outings, shared rent/utilities). Full amounts inflate personal spending.

Data model (`useBudgetStore`, persisted, new version + migration):

```ts
interface Transaction {
  // existing fields unchanged; amount = YOUR share (all budget math untouched)
  shared?: {
    totalAmount: number      // what you actually paid
    sharedWith: string       // person who owes you the difference
  }
  reimbursement?: {
    from: string             // person paying you back
  }
}
```

- TransactionModal: "Shared bill" toggle. When on: input for total paid, quick share buttons (50%, 33%, 25%, custom amount), "shared with" text input with autocomplete from previously used names. `amount` is derived as your share; `shared.totalAmount` stores the full payment.
- Triage inbox and transaction list rows show a badge (e.g. "shared · Alex") on shared transactions; the transaction list edit path reuses the modal controls.
- Income transactions get a "Reimbursement from <person>" option in the modal. Reimbursement transactions are excluded from income totals in all widgets (summary, income widget, Sankey, insights).
- IOU balances are derived, not stored: per person, sum of `(shared.totalAmount - amount)` minus sum of reimbursement amounts from that person. A negative balance renders as "overpaid" rather than being clamped.
- New "Owed to Me" widget on the Budgeting Overview tab: per-person balance list, all-time (not period-filtered), with a "Settle up" button per person. Settle-up creates a reimbursement transaction for the outstanding balance dated today so the ledger stays consistent. Widget hidden when there are no shared transactions.

## 3. Triage inbox: Clear All

- `TriageInboxWidget`: "Clear All" button next to "Accept All", opens the existing themed `ConfirmDialog` ("Discard N pending transactions? This cannot be undone."), on confirm rejects all pending transactions via the store.

## 4. Themed focus rings (white box on click)

Problem: no focus styling exists, so dark themes show the browser default white outline when clicking focusable elements.

- `src/index.css`: global rules
  - `:focus { outline: none; }`
  - `:focus-visible { outline: 2px solid var(--accent); outline-offset: 2px; }`
- Components that already draw their own focus affordance (inputs using `focus:border-accent`, `focus-within` wrappers) keep working; the global rule only affects the default outline. Spot-check interactive elements per theme during implementation.

## 5. Income Flow (Sankey) label clipping

- `SankeyWidget`: `margin.left` 10 -> 100 to match the right side.
- Node label: truncate names longer than ~14 chars with an ellipsis and a `<title>` tooltip carrying the full name.

## 6. Multi-period reports + unbudgeted spending row

Problem: Overview and Insights are locked to a single month.

- Budgeting page: the month navigator becomes a period control. Presets: This month, Last month, Last 3 months, Last 6 months, Last 12 months, Year to date. Default remains This month; the prev/next arrows and "Today" button stay and step by month when a single-month period is active (arrows hidden for multi-month presets).
- Widgets take a `{ from, to }` month-key range (inclusive) instead of a single `selectedMonth`. A shared helper produces the range from a preset and matches transactions by date prefix comparison.
- Aggregation semantics:
  - IncomeWidget, ExpenseWidget, MonthlySummaryWidget: sum over range. Month-end forecast row only shown for single current month.
  - BudgetProgressWidget: targets scale by number of months in range (target x months); pace prorating applies only within the current month.
  - SankeyWidget, CategoryTrendsWidget, AnomalyAlertsWidget: aggregate/iterate over range.
  - SpendingHeatmapWidget: single month keeps the day-grid; multi-month ranges render a month-per-cell grid (total spend per month, same color scale logic).
  - SubscriptionsWidget is range-independent (unchanged).
- Unbudgeted spending row: `BudgetProgressWidget` gains a final row "Unbudgeted" summing expense transactions in the period whose category has no target (targetAmount 0) or no category. No progress bar, amount only, error-tinted when > 0.
- Long-term (6-month/yearly) category budgets: deferred to backlog; re-evaluate after multi-period reports land.

## 7. USD to CAD reliability (Compensation)

Root cause: fx cache key includes today's date. On a new day the cache misses; on fetch failure there is no fallback, and `useCompensationDisplay` silently uses rate 1.0.

- `marketDataService.getFxRate`: when today's fetch fails and today's cache is empty, fall back to the most recent cached rate for the pair from any prior date (scan `fx` cache keys), returned with `source: 'cache'`, `stale: true`, and its original date. Only throw when no rate for the pair has ever been cached.
- `useCompensationDisplay`: expose `fxAsOf`, `fxSource`; never fabricate 1.0 silently. When no rate is available at all, `fxRate` is undefined and CAD conversion stays off with a message.
- Compensation page: the Convert to CAD toggle line shows rate provenance: "1 USD = 1.3712 CAD (as of Jul 17)" / "(manual)" / "unavailable". Next to it, an inline manual-rate NumberInput writing the same override key the Currency Converter uses (`fxKey(USD, CAD, todayKey())`), plus a clear-override affordance.
- Portfolio and Dashboard rollup FX conversions get the fallback for free via the service change.
- Unit tests: fallback-to-previous-day path, override precedence.

## 8. Luxury theme button contrast

- Replace `text-white` with `text-[var(--color-bg-primary)]` on accent-background buttons: `AddAccountModal` save button, `CompareView` line 257, `CompensationModal` RSU preset buttons (3 spots). Matches the existing app convention (gold accent + black text in luxury).

## 9. PortfolioAnalyst CSV import (Portfolio tab)

The IBKR PortfolioAnalyst export is a multi-section positional CSV (`Section,RowType,...`), not a flat holdings file; the current import falls into the generic column mapper and fails.

- New parser `src/utils/investments/ibkrPortfolioAnalyst.ts`:
  - Detection: first cell of first row is `Introduction` (plus `Header`/`Data` discriminators in column 2).
  - Extracts the curated section set:
    - Key Statistics: beginning/ending NAV, cumulative return, best/worst return, deposits and withdrawals, dividends, interest, fees, MTM, change in NAV.
    - Historical Performance Benchmark Comparison: monthly account vs SPXTR/EFA/VT return series, and the summary table (MTD/QTD/YTD/1Y/3Y/5Y/inception).
    - Allocation by Asset Class, by Sector, by Region.
    - Performance by Symbol.
    - Dividends + Projected Income.
    - Fee Summary.
    - Open Position Summary (date, symbol, description, quantity, close price, value, cost basis, unrealized P&L).
  - Ignores ESG, Concentration, Risk Measures, Corporate Actions, Trade Summary, notes.
- New persisted store `usePortfolioReportStore`: parsed report + upload timestamp; replaced on new upload.
- `PortfolioImport` / `PortfolioView`: the existing upload path tries PA detection before the flat parsers. On success:
  - Report renders as collapsible sections below holdings: stat cards (Key Statistics), benchmark comparison line chart (recharts, monthly cumulative), allocation bar lists, performance-by-symbol table, dividends/income section, fees.
  - A themed prompt offers "Also update holdings from this report?" using Open Position Summary rows (symbol, quantity, avg cost = cost basis / quantity, currency). Decline keeps holdings untouched.
- Unit tests: parser against a trimmed fixture derived from the real file (sections present, numbers extracted, malformed rows skipped).

## 10. Wheel (Options) upload: stuck on "Parsing CSV(s)"

The user's Activity Statement parses fine through `processIBKR` in Node (49 tickers), so the hang is in upload/status handling in `WheelView`.

- Rework `handleFileUpload`:
  - Track per-file completion including error paths (the current `error` callback does not increment `processed`, so any file error hangs the status forever).
  - Wrap `complete` body in try/catch; on throw, set an error status instead of leaving "Parsing CSV(s)".
  - After merge, report outcome: "Added N new trade rows (M duplicates skipped)" or a clear error. If a parse produces zero recognizable rows, say so instead of silently doing nothing.
- Verify in the browser with the real `U6147733_20260101_20260717.csv` during implementation; fix whatever the reproduction reveals if it goes beyond status handling.
