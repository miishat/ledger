## 1. PWA update flow

**Problem:** Installed PWA never surfaces the "New version available" toast; user only gets updates by visiting the website in a browser first.

**Current state:** `vite-plugin-pwa` with `registerType: 'prompt'`; `UpdateToast.tsx` checks hourly + on `visibilitychange`. There is no check at launch — a freshly opened PWA waits up to an hour (or for a focus event that may never fire in standalone mode).

**Changes:**

- In `src/components/ui/UpdateToast.tsx`: call `registration.update()` immediately in `onRegisteredSW` (launch-time check), in addition to the existing hourly + visibility checks.
- Add a **Check for Updates** button to the What's New modal footer area. States: idle → `Checking…` → `You're up to date` (transient) or `Update available` + **Refresh** button that calls `updateServiceWorker(true)`. Only one component may own `useRegisterSW`, so the registration is lifted into a small `useSWUpdate` hook called once in `Layout`, which passes state/actions down to both `UpdateToast` and `WhatsNewModal` as props.
- No workbox config changes expected; GitHub Pages serves `sw.js` fine. User data (localStorage/zustand persistence) is unaffected by SW updates.

**Non-goal:** background/push update notification.

## 2. Debt payoff — auto-calculated minimum payments

**Files:** `src/utils/finance/debtPayoff.ts`, `src/components/planner/DebtPayoffCalculator.tsx`.

**Data model:** extend `Debt`:

```ts
type DebtType = 'creditCard' | 'lineOfCredit' | 'loan'
interface Debt {
  id: string; name: string; balance: number; aprPct: number
  type: DebtType
  // loan only:
  loanMode?: 'payment' | 'term'   // "I know my payment" vs "I know my term"
  minPayment?: number              // loan, payment mode
  termYears?: number               // loan, term mode
}
```

**Minimum payment rules (decided):**

| Type | Minimum payment | Recomputed |
|---|---|---|
| Credit Card | `max(10, 3% of balance)` (payment never exceeds balance) | every simulated month |
| Line of Credit | interest-only: `balance × APR / 12` | every simulated month |
| Loan (payment mode) | user-entered fixed $/month | fixed |
| Loan (term mode) | `monthlyPayment(balance, aprPct, termYears)` from `src/utils/finance/amortization.ts`, computed from the **starting** balance | fixed |

**Engine:** `simulatePayoff` computes each debt's minimum per month via a `minPaymentFor(debt, currentBalance)` helper. The total monthly budget is **fixed at month 1**: `extraMonthly + Σ month-1 minimums`. Each month every live debt gets its *current* minimum; the remainder (extra + freed/shrunk minimums) goes to the focus debt per strategy. This preserves standard snowball/avalanche semantics and current behavior for fixed-min debts.

**UI:** each debt row gets a Type `SelectField`. Row fields by type:

- Credit Card / LOC: Name, Type, Balance, APR, **Min Payment (read-only, auto-computed, shown)**.
- Loan: Name, Type, Balance, APR, mode toggle (Payment / Term), then either Min Payment ($, editable) or Amortization (years → computed payment shown read-only).

**Migration:** stored debts (planner store persists tool inputs as JSON) lacking `type` are treated as `type: 'loan', loanMode: 'payment'` with their existing `minPayment` — done in `parseDebts`, no store version bump needed. Default example debts updated to showcase types (e.g. Credit Card + Car Loan in term mode).

**Explicitly out (decided):** biweekly payments in debt payoff.

**Edge cases:** interest-only LOC minimum never retires the debt — the sim already reports "Never (payments too low)" if the fixed budget can't cover it; with the fixed-total-budget model the extra normally retires it. CC 3% floor prevents asymptotic never-ending balances.

**Tests (minimal):** unit tests for `minPaymentFor` (3 types + $10 floor) and one `simulatePayoff` case with a declining CC minimum.

## 3. Mortgage — accelerated biweekly toggle

**Files:** `src/utils/finance/amortization.ts`, `src/components/planner/MortgageCalculator.tsx`.

- New util `acceleratedBiweeklySchedule(principal, annualRatePct, years)`: payment = `monthlyPayment(...) / 2`, period interest = `balance × (rate/100) / 26`, stepped per 2-week period until balance ≤ 0. Returns points with period index → convertible to years for the chart.
- Payment mode gets a **Monthly / Biweekly** payment-frequency toggle (same pill style as the existing Payment/Affordability toggle). Biweekly = accelerated (Canadian lender convention).
- When Biweekly: result cards show **Biweekly Payment**, **New Payoff Time** (y/m), **Interest Saved vs Monthly**. Chart shows biweekly balance curve vs monthly baseline.
- **Extra Payments section is hidden while Biweekly is selected** (stays a monthly-only feature).
- Affordability mode unaffected.

**Tests (minimal):** one unit test asserting the biweekly schedule pays off faster than monthly and saves interest for a known case.

## 5. Analysis modal shake fix

**Problem:** while typing ticker/exchange, the row's status suffixes change ("(fetching…)" ↔ "(auto)" ↔ "(manual)"), changing label min-content width and re-flowing the CSS grid — the modal visibly jitters.

**Fix in `TickerRowEditor.tsx` (and the Add Position modal's price label):** make labels layout-stable — single-line labels with `truncate` + `min-w-0` on grid cells so column widths never depend on label text length. Verify no other dynamic text (allocation-sum line already uses `invisible` to reserve space).

## 10. Mobile bottom cutoff

**Cause:** app shell uses `h-screen` (100vh); on mobile the visible viewport is shorter (URL bar), pushing page bottoms under the fixed bottom tab bar.

**Fix in `Layout.tsx`:** `h-screen` → `h-dvh` (dynamic viewport height; Tailwind v4 supports it). Audit `<main>` bottom padding: ensure it clears the 52px tab bar + `env(safe-area-inset-bottom)` (bump `pb-20` → `pb-24` if needed after real-device/emulated check).

## 11. Disclaimer (estimates only, not advice)

**Decided:** first-launch notice + persistent footer line, plain-friendly tone.

- **First-launch notice:** one-time dismissible modal (reuses modal styling) shown when a `ledger-disclaimer-ack` localStorage key is absent; "I Understand" button sets it. Existing users see it once after updating.
- **Footer line:** short persistent text "Estimates Only — Not Financial Advice" near the version link in the sidebar footer and the mobile header row 2; clicking it opens the full disclaimer text.
- **Full text (plain language, shown in the notice and reachable from the footer):**
  > Ledger's calculators and trackers produce estimates for planning purposes only. They are not financial, investment, tax, or legal advice, and may not reflect your real rates, fees, or circumstances. Verify important numbers with a licensed professional before making decisions — you are solely responsible for how you use this information.
