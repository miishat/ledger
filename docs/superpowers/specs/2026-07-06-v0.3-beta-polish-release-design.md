# v0.3-beta Polish Release — Design

Date: 2026-07-06
Status: Approved by user (brainstorming session)

## Overview

A bugfix/polish release covering 12 items across app delivery, inputs, theming,
budgeting, and investing, plus changelog/versioning infrastructure. Ships as
`v0.3-beta`.

## 1. App update delivery (PWA staleness)

**Problem:** Users keep seeing old builds (some still see long-removed
"projections" UI). The cause is the service worker from `vite-plugin-pwa`
(`registerType: 'autoUpdate'`): it serves the precached old bundle and swaps
silently on its own schedule, so users never know an update exists.
localStorage is not the cause.

**Design:**
- Switch to `registerType: 'prompt'` and use the `useRegisterSW` hook from
  `virtual:pwa-register/react`.
- When a new service worker is waiting, show a toast: "New version available"
  with a **Refresh** button. Clicking it calls `updateServiceWorker(true)`
  (skipWaiting + reload).
- Register periodic update checks: on window focus/visibilitychange and on an
  hourly interval, call the SW registration `update()` so long-lived tabs
  detect new deploys.
- Users currently stuck on old versions are picked up by their existing SW's
  update check on next load; no manual cache clearing required.

## 2. Numeric input behavior

**Problem:** Numeric fields snap to `0` when cleared; typing then appends
after the zero (`0100`). Only workaround is select-and-replace.

**Design:**
- One shared `NumberInput` component (in `src/components/ui/`). Internal state
  is a **string**: the field may be empty while focused. On blur (or Enter),
  the string is parsed and committed via `onChange(number)`; blank commits
  as 0 (or the field's configured minimum).
- Spinner arrows removed app-wide (CSS: hide `::-webkit-inner/outer-spin-button`,
  `-moz-appearance: textfield`), `inputMode="decimal"` for mobile keyboards.
- All numeric fields across Planner, Compensation, Budgeting, Investments,
  and Settings migrate to `NumberInput`.

## 3. Aurora theme dropdown readability

**Problem:** In the aurora gradient theme, dropdown menus (Planner,
Compensation, Budgeting, etc.) inherit translucent card backgrounds
(`backdrop-filter: var(--card-blur)`) and are hard to read over the gradient.

**Design:**
- Add an opaque `--dropdown-bg` design token defined per theme (aurora gets a
  solid dark surface; other themes map it to their existing surface color).
- All floating menus — custom dropdown popovers and native `select option`
  styling — use `--dropdown-bg` with no backdrop blur. Cards keep their glassy
  look; anything overlapping other content becomes opaque.

## 4. Budgeting overview layout + chart tooltip

**Problem A:** Income Flow (Sankey) takes a full row on desktop because
`SankeyWidget` hardcodes `md:col-span-2` while the page places it in a 2-col
grid next to Budget Progress.

**Fix A:** Remove the `md:col-span-2` from `SankeyWidget`'s wrapper so it
occupies one column on desktop, full width on mobile.

**Problem B:** Chart hover tooltips are unreadable in dark themes — the
tooltip box is styled via `contentStyle` but the item/label text keeps
recharts' default near-black color.

**Fix B:** Set `itemStyle` and `labelStyle` to theme text colors on the
Sankey tooltip, and audit **every** recharts `<Tooltip>` in the app for the
same gap. Prefer a shared tooltip style object/helper so all charts stay
consistent.

## 5. Analysis modal reflow while typing

**Problem:** In Investments, the New Analysis modal visibly shakes/reshapes on
every keystroke.

**Design:** Diagnose first (systematic-debugging) — likely causes are
content-driven width (auto-sizing to the widest row) and conditional elements
(validation messages, computed per-ticker rows) appearing per keystroke.
Fix by stabilizing layout: fixed modal width, reserved space (min-heights /
always-rendered placeholders) for content that appears or disappears during
typing. No keystroke may change the modal's outer dimensions.

## 6. Plan/Actual summary strip

**Problems:** Total Return is % only; the strip leaves large trailing empty
space (a lot in Plan, some in Actual).

**Design:**
- Summary strip becomes a full-width CSS grid with evenly distributed cells —
  no dead zone.
- Add **Total Return $** alongside the %.
- Plan strip: Start Date · Planned Budget · Current Value · Total Return
  ($ and %).
- Actual strip: see item 7.

## 7. Actual tab rework (investing)

**Problems:** Extra investments added after the initial one are counted in the
total but the summary still labels everything "initial". Manual initial/extra
fund inputs invite inconsistency. The lone "Planned Budget" input above the
Plan summary looks orphaned.

**Design:**
- Remove the manual initial-fund and extra-fund inputs from the Actual tab.
- Derive both values from recorded trades:
  - **Initial Investment** = total cost of trades on the earliest trade date
    in the analysis.
  - **Extra Investment** = total cost of all trades after that date.
- Actual summary strip: Start Date · Initial Investment · Extra Investment ·
  Current Value · Total Return ($ and %).
- Plan tab: remove the standalone Planned Budget input; the Planned Budget
  value inside the summary strip becomes click-to-edit (inline edit on click,
  commit on blur/Enter).
- Migration: analyses with previously entered manual fund values fall back to
  trade-derived numbers; persisted manual fund fields are dropped in a store
  version migration.

## 8. Income on the spending calendar

**Problem:** The spending heatmap/calendar renders income the same way as
spending, polluting the heat scale.

**Design:** Income no longer contributes to a day's spend total or heat color.
Income days get a small distinct green marker (dot/+$ badge); hover shows the
income amount. The calendar legend gains an entry explaining the marker.

## 9. Separate income vs expense categories

**Problem:** The transaction modal offers the same category list for income
and expense.

**Design (user-specified):** Categories come from budget setup.
- `CategoryGroup` gains `kind: 'income' | 'expense'`. Store migration marks
  the existing "Income" group (seed id `g-2`, or matched by name) as income;
  all others default to expense. New groups choose a kind in budget setup.
- The transaction modal (and triage/categorization flows that assign
  categories) filters the category picker by the transaction type: income →
  categories in income groups; expense → categories in expense groups.
- Adding a new income source or expense category in budget setup makes it
  available immediately — no separate category list to maintain.

## 10. Changelog + in-app What's New

**Design:**
- `CHANGELOG.md` at repo root in Keep-a-Changelog format: `Unreleased`,
  `0.3-beta`, `0.2-beta`, `0.1-beta` sections. `0.1-beta` and `0.2-beta` are
  reconstructed from git history; this release's items land under `0.3-beta`.
- The changelog is bundled into the app at build time (raw import).
- A **What's New** panel renders the changelog. It opens automatically once
  when the running version differs from the persisted last-seen version, and
  is always reachable from Settings.
- The current version string (e.g. `v0.3-beta`, injected at build via a Vite
  `define` from `package.json`) is displayed in the app (Settings/footer).

## 12. Budgeting mobile overflow

**Problem:** On mobile, the Budgeting page can be scrolled horizontally.

**Design:** Audit the page at ~375px width. Grid children get `min-w-0`; wide
content (transaction table, heatmap, Sankey) scrolls inside its own card
(`overflow-x-auto` on the widget body) rather than stretching the page; charts
remain bounded by `ResponsiveContainer`. Acceptance: no horizontal page
scroll at 375px on any Budgeting tab.

## Testing

- Vitest: `NumberInput` editing semantics (clear → type → blur), category
  filtering by group kind (including migration), initial/extra investment
  derivation from trades, What's New shown-once logic.
- Browser preview verification: aurora dropdown opacity, budgeting desktop
  grid and mobile overflow (375px), chart tooltips in dark themes, analysis
  modal stability while typing, summary strip layout.

## Out of scope

- Removing the PWA/offline support.
- Redesigning the budgeting calendar beyond the income marker.
- Any data-model changes beyond the migrations named above.
