# v0.7.3-beta Design

**Date:** 2026-07-21
**Status:** Approved
**Scope:** Four independent changes: annual budget cadence, compensation price precision, multi-currency holdings, portfolio report and holdings presentation.

## 1. Annual budget cadence

### Problem

`Category.targetAmount` is treated as a monthly figure everywhere. `BudgetProgressWidget` computes a multi-month target as `targetAmount * months`, and `totalMonthlyBudget` sums raw targets. There is no way to express "$2400 for vacations this year". Entering 2400 declares a $2400/month budget; entering 200 makes the month you actually book the trip look catastrophic and every other month look artificially healthy.

### Model

A category carries one target amount plus a cadence that says what period that amount covers.

```ts
// src/types/budget.ts
export type BudgetCadence = 'monthly' | 'annual';

export interface Category {
  id: string;
  groupId: string;
  name: string;
  targetAmount: number;
  /** Period targetAmount covers. Absent means 'monthly'. */
  cadence?: BudgetCadence;
}
```

The field is optional and absence means monthly, so no persist migration is required. The store stays at version 3.

Annual categories contribute `targetAmount / 12` to every monthly total (sinking-fund model) and are tracked against calendar-year YTD spend in their own section. The year window is the calendar year of `range.to`, matching the existing `ytd` period preset.

### New module: `src/utils/budget/cadence.ts`

Pure, store-free, following the pattern of `period.ts` and `sharedExpenses.ts`.

```ts
export function cadenceOf(cat: Category): BudgetCadence
export function isAnnual(cat: Category): boolean
/** Contribution to a single month's budgeted total. */
export function monthlyEquivalent(cat: Category): number
/** Full-year target. */
export function annualTarget(cat: Category): number
/** Months elapsed in `year` as of `now`; 12 for any past year, 0 for a future one. */
export function elapsedMonthsInYear(year: number, now?: Date): number
/** Amount that should have been set aside by now. */
export function setAsideExpected(annual: number, elapsedMonths: number): number
```

`setAsideExpected` is `annual * elapsedMonths / 12`.

### Changed call sites

- `budgetSelectors.totalMonthlyBudget` sums `monthlyEquivalent(cat)` rather than `cat.targetAmount`, over expense categories only, unchanged otherwise.
- `useBudgetStore.getMonthlyBudgetStats` seeds `categoryEffectiveTargets` from `monthlyEquivalent(cat)`. Reallocations continue to apply on top as absolute dollars, so zero-based and target-based math stay coherent.
- `BudgetProgressWidget` splits its category list by cadence.

### BudgetProgressWidget layout

The widget keeps its current Total line and monthly bars, then adds an Annual section below a divider, rendered only when at least one annual expense category has a target above zero.

Total line: `budgeted` continues to come from `totalMonthlyBudget * months`, which now includes annual categories at their monthly equivalent. Spend is all in-range expense spend, including spend booked against annual categories.

Monthly section: only `cadence === 'monthly'` categories, math and pace logic unchanged.

Annual section, per annual category, for the calendar year of `range.to`:

- `annual = annualTarget(cat)`
- `spentYtd` = expense transactions in that category with a date inside that calendar year, up to and including the month of `range.to`
- `elapsed = elapsedMonthsInYear(year, <the viewed month>)`, which yields the number of months the spend window covers. It must be anchored to `range.to`, never to the wall clock: anchoring to the wall clock compares a partial-year spend sum against a fuller expected contribution whenever a past month or year is viewed, so every annual category reads as falsely under pace
- `expected = setAsideExpected(annual, elapsed)`
- Bar fill is `spentYtd / annual`, clamped to 100
- Pace uses the existing thresholds: over above `expected * 1.1`, under below `expected * 0.9`, on pace between
- Copy: `Set aside $800 by now · $250 spent · $2150 left this year`

### Accepted trade-off

In the month a large annual expense is actually booked, the Total line shows an overspend, because real spend is compared against the monthly-equivalent contribution. This is inherent to the sinking-fund model and is explained by the Annual section directly beneath it. Confirmed as acceptable during design.

### CategoryManagerWidget

Beside the existing target `NumberInput`, a two-state toggle rendering `/mo` and `/yr`. Selecting a state calls `updateCategory(cat.id, { cadence })`. The toggle is shown only for expense categories; income categories keep the current monthly-only behaviour.

### Tests

- `cadence.test.ts`: every exported helper, including a past year (12 elapsed), a future year (0 elapsed) and mid-year boundaries.
- `budgetSelectors.test.ts`: `totalMonthlyBudget` over mixed-cadence categories.
- `useBudgetStore.test.ts`: `getMonthlyBudgetStats` uses monthly equivalents; reallocations still apply on top for an annual category.
- `BudgetProgressWidget.test.tsx`: annual section renders with the set-aside copy; is absent when no annual categories exist; pace label reacts to spend above and below expected.
- `CategoryManagerWidget.test.tsx`: toggling cadence persists to the store.

## 2. Compensation price precision

### Problem

`Compensation.tsx` renders the FX-converted `pkg.companyCurrentPrice` into a `NumberInput`, which displays `String(value)`. A converted price such as `245.67 * 1.3712` produces `336.8226...` with a long float tail inside a `w-24` box, so the number visibly truncates.

### Change

`NumberInput` gains one optional prop.

```ts
maxDecimals?: number
```

It affects display only. When set, the not-editing display is `value.toFixed(maxDecimals)` with trailing zeros and any trailing decimal point trimmed, so `336.82176` renders `336.822` and `100` still renders `100` rather than `100.000`. Focus seeds the edit buffer with the same trimmed string. Typing, parsing and committing are untouched, so a user who types `336.8217` still commits `336.8217` and no stored value or downstream calculation changes.

### Applied at

`maxDecimals={3}` on every FX-convertible price field:

- `src/pages/Compensation.tsx` current stock price, with the input widened from `w-24` to `w-28`
- `src/components/compensation/CompensationModal.tsx` company current price, ESPP locked-in price, and each RSU grant price
- `src/components/compensation/CompareView.tsx` equivalent price fields

The manual FX rate input keeps its 4-decimal placeholder and is not changed.

### Tests

`NumberInput.test.tsx`: renders `336.822` for `336.82176` with `maxDecimals={3}`; renders `100` not `100.000`; without the prop behaviour is byte-identical to today; typing a longer decimal while focused is preserved and committed at full precision.

## 3. Multi-currency holdings

### Problem

`toCurrency()` in `portfolioCsv.ts` is `raw === 'USD' ? 'USD' : 'CAD'`. Every IBKR position denominated in EUR, GBP, CHF, HKD or anything else is silently stored as CAD and counted at par against Canadian dollars. `Holding.currency` is typed `'USD' | 'CAD'` and `toCad()` knows exactly one rate. Separately, live quotes carry their own `currency` which the app ignores entirely, so a quote returned in a different currency than the holding's cost basis is compared directly against it.

The FX layer is already capable: `fetchFxRateRouted` sends Frankfurter-supported pairs to Frankfurter and everything else to er-api, which covers roughly 160 currencies.

### Currency list

Extend `CURRENCIES` in `src/services/marketData/types.ts` with the codes brokers actually emit: CHF, HKD, SGD, NZD, SEK, NOK, DKK, MXN, BRL, CNY, PLN, ZAR. `FRANKFURTER_CURRENCIES` is unchanged, so the additions route to er-api automatically.

### Holding type

```ts
export interface Holding {
  id: string;
  ticker: string;
  name?: string;
  exchange?: string;
  quantity: number;
  /** Per-share cost basis in the holding's own currency. */
  avgCost: number;
  /** null when the import gave a code we cannot map. */
  currency: Currency | null;
  account: string;
}
```

`null` is a first-class state meaning "unknown", not an error. Such rows display their native numbers, show a `Set currency` affordance, and are excluded from CAD totals with a visible count.

### Parsing

`toCurrency(raw)` uppercases and trims, returns the code when it is in `CURRENCIES`, and returns `null` otherwise. Both named parsers and `mapPortfolioRows` use it unchanged in shape.

### Valuation

```ts
// src/utils/investments/portfolioMetrics.ts
/** Rate from each currency to CAD. CAD is implicitly 1. */
export type FxRates = Partial<Record<Currency, number>>

export function toCad(value: number, currency: Currency | null, rates: FxRates): number | null
export interface PortfolioTotals {
  investedCad: number
  valueCad: number
  plCad: number
  plPct: number | null
  /** Holdings left out of the totals for want of a rate. */
  excludedCount: number
}
export function portfolioTotals(rows, rates: FxRates): PortfolioTotals
```

`toCad` returns the value unchanged for CAD, `value * rate` when a rate exists, and `null` for a null currency or a missing rate. `portfolioTotals` skips null results and counts them.

### Rates hook

New `src/services/marketData/useFxRates.ts`:

```ts
export function useFxRates(currencies: Currency[], to?: Currency): {
  rates: FxRates
  missing: Currency[]
  status: FetchStatus
  stale: boolean
  refresh: () => void
}
```

Built on the existing `getFxRate` service, mirroring `useFxRate`'s resolve/mounted/online/refresh structure. The currency list is normalized to a sorted deduplicated key so the effect does not re-run on array identity. `to` defaults to `'CAD'`; CAD maps to 1 without a fetch.

`PortfolioView` passes the union of currencies present across holdings plus any currency reported by a live quote.

### Quote currency

`HoldingRow` and `HoldingCard` read `live.data.value.currency`. When it differs from the holding's currency, the price is converted into the holding's currency via the cross rate `rate(quote -> CAD) / rate(holding -> CAD)` before value and P/L are computed. When the cross rate is unavailable, the row falls back to native display and is counted as excluded.

Rows report `{ price, currency }` upward through `onPrice`, so `PortfolioView` can request rates for quote currencies it has actually seen.

### Legacy data

Portfolio store bumps to version 2. The migration leaves holdings untouched (the original currency is genuinely unrecoverable from stored data) and sets `currencyReviewPending: true` when any holding exists. A dismissible banner above the holdings explains that imports made before v0.7.3 stored foreign positions as CAD and suggests re-importing or correcting rows. Dismissing clears the flag.

### Inline correction

New store action `setHoldingCurrency(id, currency: Currency | null)`. Each holding row and card carries a compact currency control (a badge that becomes a select) so a single wrong row can be fixed without re-importing an account.

### Tests

- `portfolioCsv.test.ts`: EUR, GBP, CHF and lowercase variants map through; an unknown code yields `null`; blank yields `null`.
- `portfolioMetrics.test.ts`: `toCad` per currency, missing rate, null currency; `portfolioTotals` across three currencies; `excludedCount` for unrateable rows.
- `useFxRates.test.tsx`: resolves several currencies; CAD needs no fetch; reports missing ones; stable across array identity changes.
- `usePortfolioStore.test.ts`: v1 to v2 migration sets `currencyReviewPending` only when holdings exist; `setHoldingCurrency` updates one row.
- `HoldingRow.test.tsx`: quote in a different currency is converted before P/L; missing cross rate falls back to native and flags.

## 4. Portfolio report rework and holdings polish

### Report placement

The report stays on the Portfolio tab and is wrapped in a single card that is collapsed by default, so it no longer stretches the tab on load. The header carries the title, period, expand chevron and the existing clear-report action.

### Report structure

Expanded, the order is:

1. **Summary** (open). A hero row built from the existing `Stat` component: Ending NAV, cumulative return with the delta against the primary benchmark, change in NAV, income (dividends plus interest), fees and commissions, deposits and withdrawals. Replaces today's flat ten-item grid, which gives every figure equal weight.
2. **Performance** (open). The existing growth-of-100 line chart and the benchmark summary table, unchanged in content.
3. **Allocations**. Three donut charts (asset class, sector, region) with legends showing percentage and value, replacing the three bar-lists.
4. **Contributors** (new). A horizontal bar chart of the top five contributors and bottom five detractors by `contribution`. The full `performanceBySymbol` table moves behind a `Show all N symbols` control.
5. **Income**. Dividends received and projected annual, keeping the two-column layout, with totals promoted to `Stat`.
6. **Fees**. Content unchanged, collapsed.

### Report file structure

`PortfolioReport.tsx` currently carries six concerns in 217 lines and this rework roughly doubles it. It splits into `src/components/investments/report/`:

- `PortfolioReport.tsx` (shell: collapse, header, section composition)
- `ReportSummary.tsx`, `ReportPerformance.tsx`, `ReportAllocations.tsx`, `ReportContributors.tsx`, `ReportIncome.tsx`, `ReportFees.tsx`
- `Section.tsx` (the existing collapsible primitive, extracted)
- `reportMetrics.ts`: pure functions for the growth-of-100 series, the contributors and detractors split, and the income and fee totals, with `reportMetrics.test.ts`

Charts use `recharts` and the palette from `src/utils/chartTheme.ts`, consistent with existing chart widgets.

### Holdings

- Account headers become bars carrying the account name, holding count, and a right-aligned account subtotal (value in CAD and P/L).
- Table column headers become sortable (ticker, value, P/L, allocation), defaulting to value descending.
- The ticker cell carries a currency badge chip, which is also the inline currency control from section 3.
- The allocation cell becomes a micro-bar plus percentage.
- P/L uses consistent sign colouring across table and cards.
- Mobile cards mirror the same treatment.
- A new `AllocationChart` sits above the accounts: a donut with a by-holding / by-account / by-currency toggle. Sorting and grouping logic lives in a pure, tested `allocationBreakdown` helper alongside `portfolioMetrics`.

### Tests

- `reportMetrics.test.ts`: growth-of-100 compounding, contributors and detractors split with fewer than five symbols, income and fee totals.
- Per-section render tests covering empty and populated states.
- `PortfolioView.test.tsx`: report card is collapsed on first render; account subtotals; sorting reorders rows; allocation toggle switches grouping.
- `allocationBreakdown` unit tests including a null-currency holding.

## Sequencing

The four sections are independent and can be built in any order. Recommended order, cheapest and least risky first:

1. Section 2 (price precision). One prop, one call-site sweep.
2. Section 1 (annual cadence). Additive type change, no migration.
3. Section 3 (multi-currency). Touches types, parsing, valuation, a new hook and a store migration. Section 4's currency badge and by-currency allocation depend on it.
4. Section 4 (report and holdings). Largest surface, and it consumes section 3's output.

## Out of scope

- Quarterly or custom budget cadences.
- Sinking-fund envelopes with tracked contribution transactions.
- Portfolio value history and any time-series chart of holdings, which would require snapshot persistence the app does not have.
- Manual FX rate entry per currency for holdings; the existing override mechanism is not extended here.
- Editing holdings beyond currency.
